2. **Player tile sign-up** — ✅ done. Players broadcast "I'm working on this tile" to their clan via `tile_signups` (`tileSignups.js`), separate from `tile_progress` per `ADMIN_SPEC.md`'s Privacy section. Keyed on `ign` (not the anonymous auth session id) so a player keeps control of their sign-ups across logout/login.
3. **Hide/show completed tiles** — ✅ done. Independent "Hide completed tiles" checkbox on both View and Progress tabs (`login.js`'s `visibleTiles()`), since hiding on the board doesn't necessarily mean an admin wants them hidden while marking progress too.
4. **Clan progress summary at top of board (player-facing, v1-style tracker)** — total points collected so far, tiles done out of total tiles available, and an overall progress indicator that's point-based (not just tile-count-based, since tiles carry different point values via brackets). This is the simple always-visible header stat v1 had (`app.js`'s `updateStats()`) — separate from the deeper Admin Analytics tab below.
5. **Admin Analytics tab** — ✅ done. New Admin-only tab alongside Edit/Progress on `login.html`, with three pieces: (a) **cross-clan leaderboard** ranking every clan in the event by total points, via the now-guarded `clan_totals()` RPC; (b) **own-clan breakdown by bracket** (e.g. "1/1 Hard, 1/2 Easy"), own clan only per `ADMIN_SPEC.md`'s privacy rule; (c) **own-clan cumulative points-over-time graph** (hand-rolled inline SVG, no charting library), computed from `tile_progress.completed_at`, starting from the event's `start_time_utc` (falls back to `created_at` if unset — editable via `dev.html`, Dev-only, mirrors the existing End Time editor). Verified live end-to-end. Not built: the player-facing v1-style tracker itself (see item 4 above) — that's a separate, simpler always-visible stat, still open.
6. **Dev cross-clan Admin access** — ✅ done. Each clan assigned to an event now has an "Act as Admin" link on the Dev dashboard, opening `login.html?actAsClan=<id>&actAsEvent=<id>` in a new tab. That tab mints its own independent anonymous session (via the new `act-as-clan` Edge Function, verified against the caller's real Dev session) stored in `sessionStorage` rather than the shared `localStorage` session every other tab uses — necessary because a real clan login/logout in *any* tab otherwise clobbers what every tab resolves to (Supabase persists one session per browser by default, not per tab). Shows a persistent "Acting as `<clan>` (The Dev)" banner with no Logout button (signing out would end the Dev's real session everywhere, not just that tab); tile sign-ups/progress made while acting as a clan are attributed to ign "The Dev".
7. **Lock/unlock admin pages based on event start time, with a Dev override** — partially done, 2026-07-17: `current_event_started()` (`rls.sql`) now locks item_sets/item_set_members writes to Dev-only once an admin's own event has started (both RLS-enforced and visibly reflected in `login.html`'s Edit tab — create/rename/delete/add/remove controls disappear, a "Locked" note shows). Dev's override is implicit (already bypasses every write policy via `current_is_dev()`), not a separate explicit toggle. **Still open**: this was scoped down to just item_sets for now, not a general lock on Edit/Progress/Analytics tabs or tile editing itself — revisit if the same "don't let setup-time changes happen once live" concern comes up elsewhere.

## Item bank & set builder

The original 2026-07-16 version (plain Dev-only list, hand-typed name + photo URL) was replaced 2026-07-17 with a live-search model:

- **Wiki-backed search, no bulk import** — ✅ done. Investigated community datasets (osrsbox-db-style JSON dumps) first, as originally planned, but found something better: the OSRS Wiki's own `action=bucket` API (built by the wiki's own maintainers, always current — no staleness risk the way a dataset snapshot or a one-time scrape would have). `wikiItems.js` pulls the full `infobox_item`/`infobox_bonuses` index once per session (~16.5k items, ~4MB, cached in memory) and filters client-side as you type; no server-side proxy needed (the wiki's API returns CORS headers directly). An item is cached into the local `items` table (stable id, name, photo, equipment slot) the first time anyone picks it — see `getOrCreateItemFromWiki` in `admin.js`.
- **Equipment-slot-aware items** — ✅ done. `items.equipment_slot` is populated straight from the wiki's own data (`infobox_bonuses.equipment_slot`) — no manual tagging needed.
- **Item sets are now admin-creatable, not just Dev's** — ✅ done (see `ADMIN_SPEC.md`'s Known gaps → Item bank for the full model, including the event-start lock).
- **Slot-based set builder UI** — ✅ done, 2026-07-18. Item sets (both `dev.html` for Dev and `login.html`'s admin Item Sets section) now render as a 12-cell equipment doll — the 11 in-game slots (Head; Cape/Neck/Ammo; Weapon/Body/Shield; Legs; Hands/Feet/Ring) laid out per Liel's reference, plus a 12th "Other" cell for non-equipable items (kept for now, may remove later). Each cell has its own independent live-wiki search, scoped to that slot (`slotBucketFor`/`groupBySlotBucket` in `wikiItems.js`) — reuses the same `searchPickableItems`/`resolvePickedItem` orchestration as everywhere else, just filtered per cell after the fact. 2-handed weapons show in the Weapon cell with a "(2h)" badge rather than visually merging the Weapon/Shield cells — a set is a checklist, not a single loadout, so there's no one "current" equip state to collapse into (confirmed with Liel). Login.js's admin version respects the existing event-start lock exactly like the flat version did (no search boxes/remove buttons in any cell once locked).
