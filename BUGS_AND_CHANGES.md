Bug reports and small UI/behavior tweaks that aren't new features — see `FEATURE_IDEAS.md` for the feature backlog instead.

## Open

1. **Progress tab: `complete_x_times` tiles should support a custom increment/decrement amount, not just ±1.** Requested 2026-07-19. Shape: a number input (validated positive — reject 0/negative/non-numeric) plus two buttons flanking the existing ±1 controls — a custom-subtract button to the *left* of the existing "-1" button, and a custom-add button to the *right* of the existing "+1" button — so the row reads [custom −N] [−1] [+1] [custom +N]. Not yet built.

## Monitoring (believed fixed, not fully closed out)

1. **Equipment-doll: images break, slots look too small, and clicking a slot still re-renders/feels sluggish — all three used to persist in Liel's real browser and only clear after a hard refresh (Ctrl+Shift+R).** Recorded in detail so the same fixes aren't attempted twice:

   - **Symptoms Liel actually observed**: (a) item images in the doll show as broken until a hard refresh; (b) the doll/slots render too small even after the stretch-CSS change below; (c) clicking a doll slot still visibly refreshes/redraws the whole set, feeling slow — the exact symptom the DOM-surgery fix below was meant to remove.
   - **Already tried, 2026-07-18, none of it resolved what Liel saw at the time** (do not redo these):
     - Added `referrerpolicy="no-referrer"` to every doll/item `<img>` tag, on the hotlink-block hypothesis for the broken images.
     - Changed `.doll-slot`'s `background-size` from `cover` to `100% 100%` (stretch) and raised `.doll-slot-icon`'s `max-width`/`max-height` from 65% to 88%, to fix the "tiny icon in a big tile" look.
     - Rewrote slot-selection and expand/collapse in both `dev.js` and `login.js` (builder) and the `login.js` item/set viewer modal from full-list re-renders (`renderItemSets()`/`renderItemSetsManagement()`/`renderModalNSets()`) to targeted DOM updates (`rerenderItemSetCard()` for one card only; slot-select does pure `.active`-class + detail-panel-innerHTML swaps, touching zero `<img>` elements) — specifically to stop other sets'/slots' images from being torn down and recreated on every click.
   - **Why these were believed to not be the real fix**: every one of the above was verified live via Playwright against a *freshly loaded* page (no cache) and worked correctly there — the doll rendered compact, images loaded, slot-clicks didn't touch other elements. Liel then reproduced all three symptoms anyway, and a hard refresh (which forces the browser to re-fetch every asset, bypassing cache) made all three go away at once. That combination — automated fresh-load tests pass, but only a hard refresh fixed it for a real user — pointed at **stale browser caching of `style.css`/`dev.js`/`login.js` from earlier in the session** as the actual cause, not a defect in the code changes themselves.
   - **Fix applied 2026-07-19, confirmed by Liel to resolve it (treated as believed-fixed, not fully closed — see regression risk below)**: added a `?v=1` query string to the `<link>`/`<script src>` tags in `dev.html`/`login.html`/`index.html` for `style.css`/`dev.js`/`login.js`/`app.js`. This forces a normal reload to fetch a genuinely new URL instead of potentially serving a stale cached copy, confirming the caching theory above was correct.
   - **Regression risk, worth watching for**: the `?v=` number is manually maintained — nothing enforces bumping it. If `style.css`/`dev.js`/`login.js`/`app.js` are edited again without bumping the version in the HTML file(s) that reference them, this exact bug can silently come back. **Standing rule going forward: any edit to those four files must bump `?v=` in every HTML file that references them** (`dev.html` and `login.html` both reference `style.css`; each JS file is referenced by exactly one HTML entry point). Saved to session memory so this isn't forgotten across future sessions.

The picked row is removed from the results list individually instead. Verified live end-to-end (search persisted across two picks, both landed in the DB).
7. ~~Progress tab: a doll slot should turn green once its item is marked collected, not just the item chip inside its detail panel~~ — done, 2026-07-19. New `modalDollSlotButtonHtml` (`login.js`, used only by the Progress-tab item/set modal's `renderModalNSets`) adds a `.collected` class when every member in that slot is in `progress.collectedItemIds`; `.doll-slot.collected` in `style.css` overrides the slot's background with a flat green, winning over `.occupied`'s texture by source order. Verified live: collecting turns the slot green immediately, uncollecting reverts it, an unrelated slot is unaffected.
